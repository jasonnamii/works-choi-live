고홈 — 작업 흐름과 산출 문서

컨셉 한 줄을 개발 착수 문서로 바꾸고, 결정을 채워 갱신하고, 깃에 올리기까지. 흐름 → 폴더 구조 → 문서별 설명 순서로 봅니다.

2

답변 · 로컬 전파 (핑퐁)

나

decisions.md에 답한다

열린 결정은 객관식(▶ 내 결정: A/B), 새 기능·수정는 ➕ 빈칸. PRD·features는 직접 안 고침.

고홈

"고홈 업데이트" → 로컬 전파

답한 결정이 PRD·features에 자동 반영(미정→확정), 새 기능은 ID 이어서, decisions는 ✅ 결정됨으로, changelog·버전 갱신.

🔁확인 핑퐁 — 영향 큰 결정·도출 모델을 고홈이 짚어 확인.↺ 어긋나면 decisions.md로 되돌아가 다시 답함.

산출물 디렉토리 구조

기능(메뉴) 하나 = 패키지 폴더 하나. 한 폴더에 문서 6종이 같이 생깁니다. 기능이 늘면 폴더가 나란히 늘어납니다.

# 로컬 작업본 (staging) mnt/outputs/planning/ ├─ gohome-메커니즘.html ← 지금 이 페이지 └─ gallery/ ← 기능 1개 = 패키지 1개 ├─ README.md 패키지 입구·읽는 순서 ├─ PRD.md 제품 요구서 (12섹션) ├─ features.md 기능정의 표 (15컬럼, 사람용) ├─ features.csv 기능정의 데이터 (필터·엑셀용) ├─ decisions.md 결정 답안지 + 이력 ← 내가 답을 쓰는 곳 ├─ decisions.empty.md (빈 답안지 예시) └─ changelog.md 변경 이력 # 기능이 여러 개면 ── 같은 레벨에 패키지가 늘어남 mnt/outputs/planning/ ├─ gallery/ ├─ profile/ ├─ feed/ … └─ (코퍼스 정합 시) _index.md · _registry.md · _features.csv 가 상위에 추가 # 깃 정본 (배포 후) cre8orclub/planning → docs/planning/gallery/ (위 6파일 그대로)

정본은 조직 레포 cre8orclub/planning 한 곳. 모든 기능 패키지가 docs/planning/ 아래 모입니다.

PRD.md· 제품 요구서 (개발 착수 기준)

features.md / features.csv· 기능정의 리스트
