고스트프로토콜 — 작업 메커니즘

어떤 입력이든 받아 PRD와 기능정의 리스트로 바꿔 개발자에게 그대로 넘기는 기획 핸드오프 스킬. 절차는 하나도 생략하지 않았습니다.

색 막대만 읽으면 — 핵심 흐름(0 → A → B → C → D → ↺)이 드러납니다.
회색 줄까지 보면 — 빠진 절차 없이 전체 메커니즘이 보입니다.
0 메뉴 기준선→ A 입력→ B 변환→ C 7파일 패키지→ D 검증·배포→ ↺ 반복
0메뉴(GNB) 기준선먼저
  • 현재 GNB·유틸 메뉴를 먼저 제시 — _nav.md로 한 줄 보여주고 “이 구성 맞나요?” 확인 (없으면 입력·코퍼스에서 후보 제안)
  • 모든 패키지의 메뉴 값은 이 GNB 항목에 매핑됨
  • nav_check — 월초(달 바뀜)·메뉴 변경이면 세션 시작 때 다시 환기
↓
A입력 받아 해석Phase 0~1
  • 입력 하나 — 컨셉 한 줄 · 와이어프레임 · 피그마 URL · PDF · 기존 기획서 (어느 메뉴 아래인지 함께)
  • 입력 유형 판별 · feature 이름 확정 · 대상 앱·파트 확정 (디폴트 C8 6파트)
  • 피그마 못 열어도 멈추지 않음 — 이미지·PDF로 대체
  • 묵힌 미결 환기 — 3일+ 안 닫힌 결정을 맨 위로 track_open
  • 요청 시 코퍼스 정합 — 예: cre8orclub/planning 최신 fetch·레지스트리 읽기 (처음이면 로그인 1회) / 기본: 로컬만 + “미확인” 경고
↓
BPRD·기능정의로 변환Phase 2~5
  • 제품·기능 모델 정리 — 목적 · 사용자 · 핵심 객체·행동 · 성공 상태. 본질을 한 줄로
  • 기능 추출(15컬럼) — 화면 있으면 역추출, 컨셉만이면 구조화. 행 1개 = 행동 1개
  • 정책·예외·상태 점검 — 레드플래그 → 정책·예외·경계·상태 4갈래 → 컴플라이언스·IP(법무 라우팅) → 시스템 파급
  • PRD 12섹션 작성 — 근거 없는 정책·예외는 확정 않고 decisions·미결정으로
  • 4가지를 절대 안 섞음 — 근거(화면) · 의도(컨셉) · 제안(AI) · 미결정(사람이 정할 것)
↓
C7파일 패키지 · ID 연결Phase 6
  • 한 폴더에 동시 산출 — PRD · features.md · features.csv · decisions · qa · changelog · README
  • F-ID ↔ FR-ID ↔ QA 서로 연결 (끊기면 핸드오프 실패)
  • decisions — 답안지형(객관식 A/B · ➕빈칸), 열린 미결엔 📅 올린 날짜
  • qa — 8차원: 기능 · 정책 · 상태/예외 · 디자인 완성도 · 사용자 경험(모션) · 접근성 · 성능 · 정합
  • _nav 매핑 점검 · 코퍼스 _registry·통합 CSV 갱신 · 옵션 엑셀
↓
D검증 → Git 배포Phase 7
  • 검증(validate) — 7파일 · 12섹션 · F↔FR · 금지어 결정적 점검
  • 코퍼스 사후 정합 리포트 — 모순·중복·끊긴 교차참조·다른 패키지 갱신 필요? (다른 패키지는 표시만)
  • 미결 추적 리포트 + “일정 걸까요?” 한 줄 제안
  • 커밋·푸시·PR 명령 출력 → cre8orclub/planning, 본인 계정 choinamhee (SSH alias — 개인 계정 안 건드림)
  • 개발자·디자이너가 받아 바로 착수
↓
↺닫힐 때까지 반복핸드오프 후
  • 갱신 — decisions에 답(객관식·➕) → “업데이트” → PRD·features·qa로 전파 → 다시 배포 (영향 큰 결정은 1줄 확인) → B·C·D 다시
  • 미결 추적 — 다음 세션을 열면 track_open이 3일+ 묵은 미결을 맨 위로 다시 띄움 → 0·A로
  • 정합 요청 시 — 레포 다른 기획과 대조 → 로컬 수정 → 재배포 → D로
핵심은 문서를 많이 만드는 게 아니라 빠짐없이 닫는 것입니다. 그래서 입력부터 배포·반복까지 어느 절차도 생략하지 않되, 위 색 막대만 따라 읽으면 핵심 흐름이, 회색 줄까지 보면 전체 메커니즘이 한눈에 들어오도록 두 층으로 정리했습니다.
