무슨 일이 일어나는가 — 8단계

Phase 0부터 7까지 한 번에, 멈추지 않고 흐릅니다. 대부분 자동이고, 사람 손이 필요한 곳은 검정 줄로 표시했습니다.

0
입력 해석
입력 유형 판별·정규화, 폴더명·대상 앱·GitHub 레포 확정
자동
1
컨텍스트 로드 — 고립 금지
산출 전, 조직이 GitHub에 올린 최신 기획 문서를 읽어 용어·정책·ID를 맞춥니다 (원격 기준)
자동
2
제품·기능 모델 정리
목적·사용자·핵심 객체·행동·성공 상태를 한 줄 본질로 규정
자동
3
기능 추출 (15컬럼)
화면·컨셉에서 기능 후보 역추출. 확인됨 / 추정 / 미정 3등급으로 분리
자동
4
정책·예외·상태 검증
레드플래그·4축 역검증·법무 감지 + 기존 문서와 모순 점검. 근거 없으면 확정 안 함
자동
5
PRD 작성 (16섹션)
배경·목표·시나리오·요구사항·정책·예외·상태·미결정까지 골격 고정
자동
6
동시 산출 + ID 연결
6파일 생성, 기능(F) ↔ 요구사항(FR) 상호 연결, 통합 마스터 갱신
자동
7
사후검토 (Git 대조)
기존 문서와 모순·중복·끊긴 참조 점검 → 정합성 리포트 + 커밋 명령어 출력(실행은 안 함)
자동
5단계로 보면 — 로드 → 추출 → 검증 → 결정 분리 → 패키징·사후검토. 두 참고 스킬(기능인벤토리·플래닝리뷰)을 따로 부르지 않고 내부 엔진으로 흡수했습니다.

05 /

사람은 어디서 일하는가

엔진은 초안을 완성해 멈추지 않고 넘깁니다. 판단·승인·반영은 사람의 몫입니다.

01 / decisions.md
미결정 닫기
못감-정보부족 / 못감-결정부족 / 기획 확인 필요 — 빈 정책·예외·자격·과금을 사람이 채웁니다.
02 / PRD 상태
Approved 승인
스킬은 임의로 승인하지 않습니다. Draft → Review까지만 자동, "개발 착수 가능"은 사람 확인 후입니다.
03 / Git
커밋·정합 반영
스킬은 파일만 만들고 커밋 명령어만 출력합니다. 실제 커밋과 "다른 문서 갱신 필요" 반영은 사람이 합니다.
사후검토가 "다른 패키지를 고쳐야 한다"를 찾아도 표시만 합니다 — 남의 문서를 직접 수정하지 않습니다.
06 /

통합 기능정의서는 어떻게 관리되는가

PRD는 기능·메뉴 단위로 독립입니다. 하지만 기능정의는 부분부분 올린 패키지들을 플랫폼 전체 마스터 하나로 통합합니다 — 그게 _features.csv입니다.

profile
PRD.md · features.md (독립)
F-001 · F-002 …
portfolio
PRD.md · features.md (독립)
F-001 · F-002 …
feed
PRD.md · features.md (독립)
F-001 · F-002 …
scan — 모든 패키지를 한 마스터로 통합 ▼
_features.csv
플랫폼 전체 통합 마스터. 맨 앞에 패키지·전역ID 컬럼을 붙여 부분 패키지들을 한 표로.
_registry.md · _index.md
공유 엔티티·횡단 정책·용어·ID 색인(registry) + 패키지 목록 한 줄(index).
  • ID는 패키지별로 유지 — 새 패키지도 F-001부터가 정상. 다른 패키지를 가리킬 땐 전역ID feature#F-001.
  • 새 기능은 혼자 살지 않습니다 — 기존 문서의 횡단 정책(삭제 유예·차단·세션·권한)과 모순되면 안 됩니다.
  • 최종본 기준은 항상 GitHub 원격입니다 — 로컬 작업본은 최종이 아닙니다.
