기획허브팀 온보딩

기획허브(c8-pl-hub)는 컨셉 한 줄·와이어프레임·피그마 URL·PDF·기존 기획서를 개발 착수용 PRD와 기능정의 리스트로 바꾸는 핸드오프 도구입니다. 핵심은 이번 버전에 만들 것과 만들지 않을 것을 잠그는 데 있습니다.

기획팀원 · 컨펌자 · 개발자
N°02한 바퀴를 도는 세 사람
01 기획팀원 입력을 PRD와 기능정의로 만들고, 무르익으면 운영으로 보내 출고합니다. git은 몰라도 됩니다. 작성자 / MAKER
02 컨펌자 올라온 결과를 PR 화면에서 읽고, 맞으면 버튼을 눌러 운영 레포에 반영합니다. 대표·리드의 자리입니다. 검수·승인 / REVIEWER
03 개발자 운영 레포에 등록된 PRD와 기능정의를 열어, 무엇을 만들지 작업 단위로 받아 착수합니다. 수신·구현 / BUILDER
N°03두 레포 — 연구소와 운영 [v2.7]

v2.6부터 기획은 레포 두 곳을 씁니다. 새 기획은 연구소에서 자유롭게 굴리고, 무르익으면 운영으로 승격합니다. 매 작업 첫머리에 어느 트랙인지 먼저 정합니다.

N°04전체 흐름 다섯 칸

사람이 손대는 곳은 시작(연구소 탐색)과 컨펌뿐입니다. 승격·출고·등록은 스킬과 시스템이 처리합니다.

연구소 탐색/ 승격 promote/ 운영 출고/ 컨펌 · merge/ 개발 착수
N°05단계별로 무엇을 하나
N°01 · 기획팀원 올리기
  1. 새 기획은 연구소(pre)에서 시작해 자유롭게 탐색합니다.
  2. 무르익으면 운영으로 승격합니다 — strict 검증·제출청소를 거칩니다.
  3. 운영 레포에 출고하면 스킬이 PR까지 대신 만듭니다.
출고해줘 — 이 한 마디가 실행 동의
받는 것 — PR URL 1개
다음 — URL을 컨펌자에게 전달
N°02 · 컨펌자 컨펌하기
  1. 받은 URL을 열어 PR 본문 5블록을 읽습니다.
  2. 더 볼 게 있으면 PRD.md·features.md만 훑습니다.
  3. 정할 것이 있으면 그 줄에 코멘트를 남깁니다.
Merge → 머지 전략 Squash 선택
하는 일 — 버튼 클릭 2번
다음 — 운영(main)에 자동 반영
N°03 · 개발자 받기
  1. 운영 레포 main에서 _nav(메뉴 지도)를 봅니다.
  2. _index에서 패키지 목록을 찾아 해당 기능을 엽니다.
  3. PRD.md로 범위를, features.md로 작업 단위를 받습니다.
F-ID ↔ FR-ID 연결로 추적
여는 곳 — cre8orclub/planning · main
다음 — 개발 착수
N°06"출고해줘" 뒤에서 일어나는 일

운영으로 출고할 때 기획팀원은 아래 다섯 단계를 직접 하지 않습니다. 스킬이 Bitbucket을 상대로 대신 처리하고, 결과 URL만 돌려줍니다.

N°07한 기능 = 8파일 한 묶음

운영 레포에서는 어느 패키지를 열어도 같은 세트가 나옵니다. 연구소에서는 이 8파일·12섹션을 강제하지 않습니다(탐색이라 느슨).

파일무엇을 담나주로 보는 사람
README.md패키지 첫 화면 — 무엇부터 보면 되는지전원
PRD.md제품 요구서 — 범위·정책·시나리오(기본 12섹션)컨펌·개발
features.md기능정의 리스트 — 개발 작업 단위(15컬럼)개발
features.csv필터·엑셀·자동화용 데이터(보조 정본)개발·PM
decisions.md사람이 정할 미결정만 — 관련 F·FR ID 포함컨펌·기획
qa.md합격 기준 — "그게 됐는지 보는 법"개발·QA
screens.md화면 지도 — 화면 없으면 '해당 없음' 블록디자인·개발

N°08컨펌자가 읽는 PR 본문 5블록

코드나 변경 파일을 다 읽지 않아도 됩니다. 사람 말로 쓴 다섯 블록이 판단에 필요한 전부입니다.

01
바뀐 것 한눈에
기능 몇 개·미결 몇 건, 어떤 정책이 before→after로 바뀌었나.
02
기존 기획과의 정합성 [통과 / 확인 필요]
모순·중복·연결이 하나도 없으면 통과, 있으면 ⚠️로 표시합니다.
03
컨펌자가 볼 곳
기계가 못 정하는 것 — "이건 사람이 정하셔야 합니다" 항목.
04
추천
"머지 가능 — 나머지는 머지 후 보완" 또는 "보류 — 무엇부터 정하라".
05
게이트 · 스킬이 자동 기록
검증·폴더링·원격 대조 PASS 스탬프. 사람이 쓰지 않습니다.
N°09미결정의 네 가지 상태

PM 워룸과 같은 프레임입니다. 채움이 진할수록 착수를 막는 강도가 큽니다.

CLEAR가는중정보·결정이 다 있어 그대로 진행합니다.
REVIEW기획 확인 필요방향을 한 번 더 확인한 뒤 진행합니다.
BLOCK · INFO못감 — 정보부족빈칸이 채워져야 출시를 막지 않습니다.
BLOCK · DECISION못감 — 결정부족사람이 골라야 다음으로 넘어갑니다.
N°10처음 오신 분이 꼭 기억할 다섯 가지
신규는 연구소에서
새 기획은 기본으로 연구소(pre)에서 시작합니다. "운영으로 / 개발 핸드오프" 신호가 있을 때만 운영에서 바로 엽니다.
운영은 무르익은 것만
운영(planning)은 strict 검증과 제출청소를 통과한 외부 공유본만 받습니다. 정화는 승격 시점에 일어납니다.
머지 ≠ 기획 확정
Merge는 운영(main)에 반영하는 일입니다. 기획 확정(approved) 표시는 남은 미결정을 정한 뒤 갱신을 한 번 더 돌릴 때 붙습니다.
자동 머지는 없다
스킬은 PR 생성까지만 합니다. 컨펌자가 버튼을 직접 누르지 않으면 어떤 것도 main에 올라가지 않습니다.
한 패키지 = 열린 PR 1개
같은 기능을 다시 출고하면 새 PR을 만들지 않고, 열려 있던 그 PR에 커밋을 더합니다. 머지는 Squash로 main에 커밋 한 개만 남깁니다.
N°11미결정을 확정하는 길

PR 위에서 직접 고치지 않습니다. 결정은 decisions.md를 거쳐 한 바퀴 더 돌 때 공식 결정으로 반영됩니다.

컨펌자가 PR 줄에 코멘트› 기획팀원이 decisions.md에 답 기입› "기획허브 갱신" 재실행› 상태 approved
N°12처음 보는 말 풀이
연구소 · pre
기획팀 내부 탐색 레포(cre8orclub/pre-planning). 월별 폴더에서 느슨하게 굴립니다. 신규 기획의 기본 출발점입니다.
운영 · operational
개발 핸드오프 정본 레포(cre8orclub/planning). GNB flat·strict. 무르익은 기획만 들어갑니다.
승격 · promote
연구소 단위를 strict 검증과 제출청소로 정화한 뒤 운영으로 옮기는 일입니다.
PRD
제품 요구서. 만들 것과 만들지 않을 것을 잠그는 개발 착수 기준 문서입니다. DB·API·픽셀 지시는 쓰지 않습니다.
features
기능정의 리스트. 개발자가 받을 작업 단위를 행마다 하나씩 적은 15컬럼 표입니다.
F-ID · FR-ID
기능(F)과 요구사항(FR)을 서로 가리키게 묶는 연결 번호입니다(F-001 ↔ FR-001).
네 갈래 분리
이번 버전에 안 넣는 것을 candidate(다음)·required-gap(정보 빈칸)·external(다른 패키지)·decision(사람 결정)으로 나눕니다.
GNB · _nav
앱 메뉴 지도. 운영 레포 모든 패키지의 대전제이고, 각 기능이 어느 메뉴에 붙는지 여기에 매핑됩니다.
Squash 머지
여러 커밋을 main에 하나로 합쳐 올리는 머지 방식입니다. 되돌릴 때도 커밋 하나만 보면 됩니다.
