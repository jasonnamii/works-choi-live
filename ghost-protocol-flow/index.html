how to read

전체 흐름은 두 구간으로 봅니다 — 최초 생성과 반복 갱신

이 문서는 처음 실행할 때 한 번 통과하는 생성 파이프라인(Zone A)과, 이후 결정을 채워 패키지를 갱신하는 반복 운영 루프(Zone B)로 나뉩니다. 분기, 루프, 피드백은 모두 선으로 표시해 실행 흐름을 한눈에 따라가도록 구성했습니다.

모든 단계에 적용되는 다섯 가지 기준

이 다섯 가지는 특정 Phase의 세부 규칙이 아니라, 전체 파이프라인을 관통하는 운영 기준입니다. 각 단계의 판단과 산출물은 결국 이 기준을 지키기 위해 설계됩니다.

rule 4 · 핵심
근거·의도·제안·미결정 분리
화면에서 본 것=근거, 컨셉=의도, AI 보강=제안, 사람이 정할 것=미결정. 섞이면 개발자가 확정으로 오해합니다.
rule 1·6
PRD와 기능정의는 항상 함께·ID로 연결
하나만 만들면 실패. F-ID ↔ FR-ID가 서로를 가리켜야 핸드오프가 완성됩니다.
rule 2
구현 지시서가 아닌 착수 기준 문서
DB 구조·API 엔드포인트·픽셀 UI 금지. 목표는 긴 문서가 아니라 최소 완결성입니다.
rule · 멈춤 없음
입력이 부족해도 멈추지 않음
컨셉 한 줄이라도 진행하고, 빈 곳은 미결정으로 남깁니다. "정보 더 달라"로 멈추지 않습니다.
rule 9 · 루프
미결정은 정해질 때까지 추적
올렸다고 끝이 아니라 정해져야 끝. 📅 올린 날짜를 달고 세션마다 다시 환기합니다.
zone a

zone b

핵심 반복 액션 — 결정을 채우며 매듭지어 가는 루프

이 패키지는 한 번 생성하고 끝나는 문서가 아닙니다. 사람이 미결정을 확정하고 기능을 추가하면서 여러 번 갱신됩니다. 특히 컨셉 초안에서 시작한 경우에는 미결정이 많기 때문에 갱신 루프가 중요합니다. 갱신의 입구는 별도 파일이 아니라 decisions.md 답안지입니다. 결정을 적는 칸이 곧 다음 업데이트의 입력값이 됩니다.

update mode · "올렸다고 끝이 아니라 정해져야 끝" — 미결이 0이 될 때까지 반복

failure modes · 빠지기 쉬운 함정

이 흐름이 막아주는 함정 열 가지

✗반쪽 산출
PRD만 또는 기능 리스트만. → 항상 둘 다 + ID 연결
✗오버스펙
DB·API·픽셀까지 지시해 구현 지시서로 변질. → 착수 기준에 머물고 후보 수준만
✗검증 엔진 욕심
모든 기능을 위험 검토 문서처럼 길게. → 점검표는 내부 도구, 산출물엔 쉬운 말만
✗요소 나열
버튼·아이콘·텍스트를 행으로. → 행 1개 = 행동/시스템 동작 1개
✗멈춤
입력 부족하다고 "정보 더 달라"로 정지. → 한 줄이라도 실행, 미결정으로
✗근거 혼동
화면과 AI 추정을 섞음. → 근거·의도·제안·미결정 분리
✗정책 임의 확정
화면이나 문서에 없는 한도, 과금, 정책을 확정처럼 쓰는 상태입니다. → 미정 또는 제안으로 분리
✗R번호 누출
레드플래그 R번호·엔진어가 본문에. → "꼭 정할 것"으로 번역, 1~3개만
✗미결정 흩뿌림
미정을 PRD 곳곳에 단정처럼. → §12·decisions.md에 모으고 ID
✗코퍼스 강제
요청도 없는데 매번 레포를 물음. → 요청 시 활성화하는 선택 레이어
